# Task 52 — Flag known Profanity vanity-prefix patterns on import

**Status:** Not taken
**Owner:** Mobile (mobile-app)
**Spec reference:** wallet-security-vulnerabilities-spec.md TWV-2026-040, §7, §9

## Why this matters

Wintermute lost ~$160M in Sep 2022 because its gas-optimised hot-wallet
address was generated by `johguse/profanity`, which used a 32-bit seed —
brute-forceable on a consumer GPU in hours. Any user who imports a
vanity address produced by Profanity (or sibling tools with the same
entropy class) is effectively running with a public key. A one-time
import-side heuristic is cheap to ship and would have stopped
Wintermute's use of that address for production flow.

## Scope

Import-path mitigation in `services/walletService.ts`:

- Add a `checkVanityPrefixRisk(address: string)` helper that returns one
  of `{ok: true}` or `{ok: false, reason: "profanity-class-prefix"}`.
  The heuristic flags addresses whose hex prefix matches a curated list
  of observed brute-forceable vanity patterns (e.g., long runs of
  leading zeros, prefixes that appear in documented Profanity exploit
  post-mortems).
- Wire the check into the wallet import flow (seed-phrase import and
  private-key import paths). On flag: show a critical, non-dismissible
  warning screen ("This address may have been generated by a tool
  whose entropy is brute-forceable. Consider migrating funds to a
  freshly generated wallet.") with a link to the Wintermute write-up.
- Check runs again when a watched address is added (if watch-only is
  ever supported), for informational display only.
- Heuristic list lives as a JSON constant under `constants/` with a
  dated source comment; updating it is a code PR, not a remote fetch,
  so the check is offline.

Additionally, add a pre-implementation note: if a "choose your address
prefix" feature is ever built, it MUST use `expo-crypto`
`getRandomBytesAsync(32)` per attempt and never seed-roll a PRNG. Link
TWV-2026-002 as the entropy source rule.

## Rules (non-negotiable)

- Warning is non-dismissible on the import screen; the user must
  acknowledge with a checkbox before proceeding.
- False positives are acceptable; false negatives are not — err on the
  side of flagging suspicious prefixes and let the user override.
- Check is offline (no remote lookup); the heuristic list is shipped in
  the bundle.
- No vanity-prefix generation feature ships without a CSPRNG-based
  implementation reviewed against this task.

## Acceptance

- [ ] `checkVanityPrefixRisk` helper implemented with unit tests
      covering known-bad prefixes from the Profanity corpus plus a
      sample of benign addresses.
- [ ] Import flow shows the warning; dismissal requires explicit
      acknowledgement.
- [ ] Heuristic JSON constant present with dated source attribution.
- [ ] Pre-implementation note about CSPRNG for future vanity features
      linked from the design docs.
- [ ] `pnpm check:syntax` passes.

## Out of scope

- A live "vanity address" generation feature.
- Bulk-scanning historical transactions for vanity-address counterparty
  risk.
- Integration with external threat feeds; heuristic stays offline.
